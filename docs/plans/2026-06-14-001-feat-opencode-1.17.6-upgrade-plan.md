# Upgrade OpenCode to 1.17.6

## Overview

Move the project to OpenCode 1.17.6, keeping the existing three carried patches. This PR bumps everything that takes a literal version *now*: `@opencode-ai/sdk` (root + runtime), `FALLBACK_VERSION` (stock), the harness `base_version`, the Renovate cap, the clonedeps pin, and version-string tests. It deliberately does **not** hand-edit `DEFAULT_OPENCODE_VERSION` or the workspace Dockerfile `ARG OPENCODE_VERSION` — those two are owned by the `sync-default-version` job, which sets them to the real `1.17.6+harness.<sha>` after the 1.17.6 harness release publishes (the `<sha>` is the LLM-merge integration commit, unknowable until that release runs). Nothing ships from `main` directly; the action/gateway only ship when the release PR (`next`→`release`) merges, by which point the harness release + the `sync-default-version` bump have landed. Sequence: merge this PR → dispatch the 1.17.6 harness release → merge the auto-generated `sync-default-version` PR → v0.64.0 ships with `DEFAULT_OPENCODE_VERSION` pointing at the published 1.17.6 harness build.

## Problem Frame

The action/runtime and harness are pinned to OpenCode 1.17.3 (`1.17.3+harness.94c10df9`). Upstream is at 1.17.6. Research (librarian + explorer, 2026-06-14) found 1.17.4–1.17.6 are mostly MCP/v2-API additions; 1.17.6 itself is a single MCP-compatibility fix. SDK 1.17.6 is published; release assets (musl + glibc + darwin) are intact; Bun is unchanged at 1.3.14; all three carried PRs (#19961, #31859, #31638) remain open upstream (none merged into ≤1.17.6). The recommendation is **GO-WITH-CAUTION**, the caution being v2 `session.wait()` — the 1.17.3 bump silently changed its `directory` parameter, and the 1.17.4–1.17.6 v2 session-API behavior could not be fully verified from release notes alone.

## Key Technical Decisions

- **One upgrade PR + the standard release/sync-default-version flow.** This PR bumps SDK/FALLBACK/harness-base/cap/clonedeps together; `DEFAULT_OPENCODE_VERSION` and the workspace ARG are flipped to the real `1.17.6+harness.<sha>` by `sync-default-version` after the harness release. This is the established mechanism (1.17.3/C2 used it), not a new two-phase invention — and it avoids ever pinning the default to a release that does not exist yet.
- **Keep all three carries.** Re-verified open upstream; none landed in ≤1.17.6.
- **Workspace ARG advances via `sync-default-version`, not hand-edited here.** The coupled-bump job already updates the Dockerfile ARG when the harness release publishes; hand-editing it in this PR would point at a 1.17.6 harness release that does not exist yet.
- **Live probe is the gate, plus an explicit v2 `session.wait` check.** Builds/tests passing is necessary but not sufficient; the real harness path and the one historically fragile v2 call must be exercised against a real 1.17.6 server before merge.
- **No version literals in comments** (standing rule); update only values/rules and the clonedeps/AGENTS metadata.

## System-Wide Impact

- **Interaction graph:** the SDK underpins `executeOpenCode` → `sendPromptToSession` → `processEventStream`/`retry.ts`; the only version-shape-sensitive call is v2 `session.wait` (Unit 4 covers it).
- **Error propagation:** unchanged — the streaming/poll/wait error paths are version-agnostic except the wait route shape.
- **API surface parity:** action and runtime both consume the SDK; both pins move together (Unit 1).
- **Integration coverage:** the live probe (Unit 4) is the cross-layer proof unit tests cannot give for SSE + LLM behavior.
- **Unchanged invariants:** carry set, harness distribution shape, workspace-via-sync-default-version coupling, stock-download v-prefix behavior — none change here.

| SSE streaming regression (1.14.42-class) | Unit 4 live probe is the merge gate, not green builds |

Run the probe against a 1.17.6 server built from the same base+carries the release will produce |

## Documentation / Operational Notes

- After this PR merges: dispatch a 1.17.6 harness release (separate, standard runbook) **promptly**, then merge the `sync-default-version` coupled-bump PR that advances `DEFAULT_OPENCODE_VERSION` + the workspace ARG to `1.17.6+harness.<sha>`. Prompt sequencing matters because CI/dogfood workflows run the action from `main` via `uses: ./`, so between this merge and the sync-default bump, those runs pair the 1.17.6 SDK with the still-pinned 1.17.3 harness binary. That pairing is safe — the v1 SDK surface is identical across 1.17.3→1.17.6 (all changes are in the v2 module) and the v2 `session.wait` route degrades gracefully via the poll watchdog — but the skew window should be kept short.
