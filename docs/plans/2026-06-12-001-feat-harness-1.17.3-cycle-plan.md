# feat: Harness 1.17.3 cycle

## Overview

Upgrade the `@fro.bot/harness` patched-OpenCode build from base 1.16.0 to 1.17.3, carry three upstream PRs, and **eliminate the version-source drift** that caused base-version bumps to be silently missed. The drift fix is the structural heart of this change; the version bump and carries ride on top.

## Problem Frame

`packages/harness/src/base-version.ts` (`HARNESS_BASE_VERSION`) is **vestigial** — no TS code reads it. `packages/harness/harness.config.json` `base_version` is the sole authoritative source: `integrate-command.ts:209-234` reads it → `integrate.ts:299-301` makes the upstream checkout tag `v${baseVersion}` → `provenance.ts` uses it. Renovate's customManager (`.github/renovate.json5:32-40`) tracks **`base-version.ts` only** — the file nothing consumes. So Renovate bumped a dead sentinel while the real source went untracked. The drift entered at `cdcd079` (#786's 1.16.0 rebase updated the config, left the sentinel at 1.15.13). No cross-file drift-guard test covers base version (only bun-version), and the `versioned-tool` skill inventory omits the harness base-version site entirely.

## Requirements Trace

- R1. `harness.config.json` `base_version` = `1.17.3`; `integrationRefs` = the 3 carried PR full URLs with metadata.
- R2. `base-version.ts` deleted; Renovate customManager retargeted at `harness.config.json` `base_version`; bun rule untouched.
- R3. A cross-file drift-guard test added that proves: (a) Renovate actually tracks the new location (applies the real matchStrings regex to the real config and asserts a match), (b) every base-version literal across `harness.config.json`, `provenance.ts`, and `harness-release.yaml` agrees with the canonical config value, and (c) no reintroduced sentinel exists — non-vacuous (probe-and-fail verified).
- R4. The `versioned-tool` skill inventory updated to include the harness base-version site (closes the skill-coverage gap that contributed to the drift).
- R5. The full harness release pipeline (integrate → build → publish) validated by a dry-run before any real publish; dry-run proves the 1.17.3 + 3-carry path including diff-scope assertion that each carry's target files appear in the integration diff.
- R6. Real publish only after dry-run proves the path; post-publish runtime smoke confirms the published binary runs a real session before declaring success.

## Key Technical Decisions

- **Version-source fix = Option A (eliminate the duplicate).** Delete `base-version.ts`; retarget Renovate's customManager at `harness.config.json` `base_version`. One source of truth, matching the project's `versioned-tool` single-source invariant. PR #866 (Renovate bump of the dead sentinel) becomes moot → close it.
- **Carry set = all 3 survivors** (policy budget is 1-3): `anomalyco/opencode#19961` (session system/messages transform ordering), `#31859` (share plugin client runtime), `#31638` (avoid full history hydration after compaction). Dropped: `#31613` (below cutoff), `#31873` (closed, unmerged, irrelevant). Caveat accepted: all 3 are open + `mergeable_state: blocked` (not merged-to-dev) → higher rebase-maintenance risk; each gets a documented drop-condition.
- **`integrationRefs` must be full GitHub PR URLs.** Verified at `packages/harness/src/sources.ts:34-53`: `parseSource` treats any string NOT starting with `https://github.com/` as a local branch name on the source repo — so a bare `owner/repo#N` would fetch `refs/heads/owner/repo#N` (garbage) and break integrate. The `owner/repo#N` form is only the rendered label in logs/prompt, never the config value. `harness.config.json` currently has no example to copy.

## Carry Refs

**CRITICAL ref format (verified at `packages/harness/src/sources.ts:34-53`):** `integrationRefs` entries must be the **full GitHub PR URL** (`https://github.com/owner/repo/pull/N`). `parseSource` treats any string NOT starting with `https://github.com/` as a *local branch name* on the source repo — so a bare `owner/repo#N` would fetch `refs/heads/owner/repo#N` (garbage) and break integrate. The `owner/repo#N` form is only the rendered LABEL in logs/prompt, never the config value. `harness.config.json` currently has no example to copy.

## Known Limitations (accepted, not fixed this cycle)

- **Per-ref provenance SHA (ADV-7):** `integrate.ts:391-398` assigns every carry the same `resolvedSha` (the integration commit), not each PR's upstream tip. `harness patches` will show 3 identical SHAs. Documented upstream-of-us simplification; not release-blocking. Track as a future harness improvement.
- **Partial semantic mis-merge (ADV-3):** with 3 simultaneous carries, a building-but-behaviorally-wrong merge of overlapping session PRs (#19961/#31638) cannot be auto-isolated. Accepted per the explicit 3-carry decision; recovery is bisect-via-1.17.4. The diff-scope assertion + runtime smoke reduce but do not eliminate this.
- **`prompt.txt` "fix it and continue":** the merge agent absorbs errors and the workflow only sees the `opencode run` exit code. This is why the diff-scope assertion (out-of-band verification of what actually changed) is required rather than trusting the merge's exit 0.
